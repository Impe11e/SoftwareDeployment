## 1. Опис архітектури та кроки реалізації

**Крок 1: Контейнеризація веб-застосунку (Dockerfile)**  

Спочатку я підготувала Dockerfile для свого застосунку. Я використала багатоетапну збірку (якщо застосовувався distroless або чистий образ), щоб мінімізувати розмір контейнера та підвищити безпеку.

**Крок 2: Конфігурація середовища та гнучкість налаштувань (config.docker.json)**

Для роботи в Docker я створила окремий файл конфігурації config.docker.json. Це було зроблено з двох важливих причин:

- Специфіка Docker-мережі: У цьому файлі я змінила хост бази даних з localhost на db. Оскільки сервіси працюють в ізольованій мережі, вони не бачать один одного через локальний хост. Використання імені сервісу db дозволяє Docker DNS автоматично направляти запити на правильну IP-адресу контейнера бази даних.

- Збереження сумісності з Лабораторною №1: Я свідомо винесла ці налаштування в окремий файл, щоб не руйнувати логіку роботи першої лабораторної роботи. Це дозволяє проекту залишатися гнучким:

  - При локальному запуску (як у Лаб №1) застосунок продовжує використовувати стандартний config.json з налаштуваннями для локальної машини.

  - При запуску в Docker Compose через механізм volumes файл config.docker.json підміняє стандартний конфіг всередині контейнера.

  Таким чином, я забезпечила стабільну роботу проекту в обох середовищах без необхідності щоразу редагувати вихідний код.

**Крок 3: Опис інфраструктури (docker-compose.yml)**  

Я об'єднала всі сервіси в одному файлі. Основні реалізовані вимоги:

**Ізоляція мережі:**  
Я відмовилася від мережі за замовчуванням (default), щоб уникнути конфліктів з іншими проектами. Для цього я визначила власну мережу:

```yaml
networks:
  lab2_net:
    driver: bridge
```

Кожен сервіс (db, web, nginx) було примусово підключено до цієї мережі.

**Збереження даних (Persistence):**  
Щоб дані БД не зникали, я використала іменовані томи (Volumes). Я примонтувала том db_data до внутрішньої папки Postgres:

```yaml
volumes:
  db_data:/var/lib/postgresql/data  
```

Це гарантує, що дані фізично зберігаються на диску хост-системи, а не всередині тимчасового шару контейнера.

**Reverse Proxy:**  
Nginx було налаштовано як вхідну точку на порту 8080, яка перенаправляє запити на порт 3000 контейнера web.

## 2. Як реалізовано вимоги щодо стійкості даних  

Завдяки використанню volumes, дані БД виживають у наступних сценаріях:

- **Перезапуск контейнерів:** Docker просто перепідключає існуючий том.

- **Видалення контейнерів (docker compose down):** Команда видаляє лише процеси, але іменований том db_data залишається на диску.

- **Перезавантаження системи:** Оскільки дані лежать на фізичному диску, після ввімкнення ОС Docker автоматично піднімає контейнери (завдяки restart: always) і підключає старі дані.

- **Видалення Docker:** Навіть при видаленні самого демона Docker, вміст директорії /var/lib/docker/volumes/ зазвичай залишається в системі.

## 3. Процес тестування та перевірки  

Я перевірила роботу системи наступним чином:

- Запустила проект: 

```bash
docker compose up -d.
```

- Додала дані через API:

```bash
curl -X POST http://localhost:8080/notes -H "Content-Type: application/json" -d '{"title": "Nu tipa test", "content": "Postavte maximum please"}'
```

- Перевірила наявність запису безпосередньо в БД:

```bash
docker exec -it lab2_db psql -U app -d mywebapp_db -c "SELECT * FROM notes;".
```

- Виконала повну зупинку та видалення: 

```yaml
docker compose down (docker compose restart).
```

- Знову підняла сервіси та переконалася, що записи в таблиці залишилися на місці.

## 4. Труднощі під час виконання  

Під час роботи я зіткнулася з одною значною проблемою:

**Race Condition:** Застосунок намагався підключитися до бази швидше, ніж вона встигала ініціалізуватися. Я вирішила це додаванням *healthcheck* для БД та умови *condition: service_healthy* у *depends_on* для веб-сервісу.
